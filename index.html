Understanding the Users

Five segments. Five different jobs. One platform.

Using Advanced Jobs-to-be-Done (AJTBD) analysis, I identified five user segments that would benefit most from a context-aware support agent — then selected the highest-impact one for the POC.

Segment 1: Autopilot Investor with a Broken Routine Selected for POC

Core Job: "When my savings plan didn't execute and I don't know why, I want to understand what happened and fix it immediately."
Big Job: "I want my money to grow on autopilot so I can focus on my life."
Who: Working professionals (25-45), set-and-forget investors, time-poor. ~270K users/year affected.
Highest escalation volumeNeeds account dataAction-capable fixPlatform-reusable

Segment 2: Active Trader Blocked by Execution Problem

Core Job: "When my order failed, I want to know why and fix it before the market moves."
Who: Self-directed traders (30-55), PRIME+ subscribers, time-critical issues. ~90K/year.

Segment 3: New Investor Overwhelmed by First Steps

Core Job: "When I'm stuck during onboarding, I want step-by-step guidance for my specific situation."
Who: First-time investors (20-35), triggered by social media. ~120K/year during onboarding.

Segment 4: Tax Season Client Needing Documentation

Core Job: "When tax season comes, I want to find and understand my tax documents without confusion."
Who: German investors during filing season (Jan-May). Massive seasonal spike. ~400K/year.

Segment 5: Client Considering Leaving

Core Job: "When I've had multiple bad experiences, I want my issue resolved properly to restore trust."
Who: Pre-churn clients with repeat contacts. ~30K/year. Highest LTV at risk.

Why Segment 1 was selected

This segment has the highest volume of currently-escalated requests (FAQ bot can't access savings plan data), directly maps to the "client-specific data + action execution" vision described by the team, and the client context layer built for this use case is reusable for all other segments.

Why the top risks are addressable

Compliance (Score 15): Scalable has already started feeding client-specific data to the LLM (confirmed in interview). The expansion to savings plan data is incremental, not a paradigm shift. Start with read-only access as a compliant first step.

Hallucination (Score 15): Unlike free-text retrieval, savings plan data is structured (amount, date, status, cash balance). Structured data injection has dramatically lower hallucination risk than unstructured RAG. Add confidence thresholds and auto-escalation as safety nets.

Task 2 · Evaluation Framework

How to measure AI quality before and after release.

Three evaluation layers — offline testing, online monitoring, and compliance-specific checks — aligned with NIST AI RMF[8] Measure function and BaFin's Dec 2025 guidance on AI/ICT risk management[4].

1

Offline Evaluation (Pre-Release)

Golden Test Set

500+ labeled support cases with known-correct answers. Run on every model or prompt change. Refreshed monthly from production trace analysis. All prompts version-controlled; regressions block deployment.

Metrics

Answer accuracy (F1), hallucination rate, retrieval quality via RAGAS[2] (context relevance, faithfulness, answer relevancy), action safety.

Hallucination Detection

Three-signal approach: (1) Structured data comparison — flagged if agent says €X but source says €Y. (2) NLI entailment checking[10] — catches fabricated procedures. (3) Policy-compliance validation against regulatory ruleset.

LLM-as-Judge

Separate evaluation model scores each response on faithfulness, helpfulness, safety. Calibrated quarterly against human reviewers to correct for known biases[10].

2

Online Evaluation (In Production)

Resolution Rate

% of conversations resolved without human escalation. Current: 70% (per interview with Sebastian). Target: 85%.

Agent Quality Score

Inspired by Intercom's CX Score[9]. Single north-star metric: AQS = 0.4×Res + 0.3×(1−Hall) + 0.2×CSAT + 0.1×Actions

CSAT

Post-conversation satisfaction (thumbs up/down + optional comment). Supplemented by ML-based sentiment analysis for full coverage.

Action Completion

% of AI-proposed actions that clients confirm and complete. Target: >40%.

Drift Monitoring

Weekly rolling average of all key metrics. Alert if any degrades >2pp from 30-day baseline. Critical for seasonal question distribution shifts.

Trace Analysis

Weekly review of failed/escalated conversations by PM + Support Lead. Findings feed into golden test set refresh and prompt iteration backlog.

3

Compliance-Specific Evaluation

Data Leakage Score

Automated checks that no PII or financial data from Client A appears in Client B's conversation.

Regulatory Boundary

% of responses correctly refusing investment advice. Target: 100%. Tested with adversarial prompts ("What stocks should I buy?").

Bias Testing

Differential response quality across demographics (language, account size, age). Flagged if resolution rate varies >5% between cohorts.

Escalation Path

Violation detected → circuit breaker (topic disabled) → DPO incident report within 1h → root cause within 24h → fix deployed before topic re-enabled.

Task 2 · Guardrails

Five defense layers against hallucination and data leakage.

Compliance is a design constraint, not a gate at the end. Every response passes through input sanitization, processing constraints, output classification, immutable audit logging, and regulatory controls — aligned with DORA[6], the EU AI Act[5], and GDPR Article 22[7].

1

Input Guardrails

  • Prompt injection detection — block adversarial inputs before they reach the LLM
  • Topic classification — route to support, investment (Insights), or reject
  • PII masking — name, email, IBAN, tax ID masked before LLM. Portfolio data sent as structured context under data minimization
  • Rate limiting per session
2

Processing Guardrails

  • Hard constraints — no investment advice, no other clients' data, no actions without confirmation
  • Retrieval scope — model accesses only authenticated, consenting client data
  • Action allowlist — only pre-approved operations (reschedule, adjust, escalate)
  • Third-party risk — LLM provider governed under DORA ICT framework[4]. DPA ensures no client data used for training
  • Fallback — LLM unavailable → FAQ bot. Low confidence 3x → human escalation with full context
3

Output Guardrails

  • Response classifier — checks every output for hallucinated facts, investment advice, cross-client leakage
  • Confidence threshold — below 0.80 triggers automatic escalation to human agent
  • Mandatory disclaimers on any financial topic
4

Observability & Audit

  • Immutable trace logging — full conversation and decision log (append-only)
  • Real-time dashboard — hallucination spikes, escalation surges, latency anomalies
  • DORA incident management — failures classified via ICT incident taxonomy[6]
  • Business continuity — includes AI system unavailability scenarios
5

Regulatory Compliance

  • EU AI Act — Preliminary classification: limited-risk (support tool, not autonomous decision-maker). Client confirmation maintains human agency. Per Art. 6(3)[5]
  • GDPR — DPIA before deployment. Opt-out available. Client confirms all account modifications — satisfying Art. 22[7]
  • BaFin — Quarterly model audit per AI/ICT guidance[4]. Every decision traceable to source data via compliance trace log

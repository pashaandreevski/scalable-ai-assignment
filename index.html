Understanding the Users

Five segments. Five different jobs. One platform.

Using Advanced Jobs-to-be-Done (AJTBD) analysis, I identified five user segments that would benefit most from a context-aware support agent — then selected the highest-impact one for the POC.

Segment 1: Autopilot Investor with a Broken Routine Selected for POC

Core Job: "When my savings plan didn't execute and I don't know why, I want to understand what happened and fix it immediately."
Big Job: "I want my money to grow on autopilot so I can focus on my life."
Who: Working professionals (25-45), set-and-forget investors, time-poor. ~270K users/year affected.
Highest escalation volumeNeeds account dataAction-capable fixPlatform-reusable

Segment 2: Active Trader Blocked by Execution Problem

Core Job: "When my order failed, I want to know why and fix it before the market moves."
Who: Self-directed traders (30-55), PRIME+ subscribers, time-critical issues. ~90K/year.

Segment 3: New Investor Overwhelmed by First Steps

Core Job: "When I'm stuck during onboarding, I want step-by-step guidance for my specific situation."
Who: First-time investors (20-35), triggered by social media. ~120K/year during onboarding.

Segment 4: Tax Season Client Needing Documentation

Core Job: "When tax season comes, I want to find and understand my tax documents without confusion."
Who: German investors during filing season (Jan-May). Massive seasonal spike. ~400K/year.

Segment 5: Client Considering Leaving

Core Job: "When I've had multiple bad experiences, I want my issue resolved properly to restore trust."
Who: Pre-churn clients with repeat contacts. ~30K/year. Highest LTV at risk.

Why Segment 1 was selected

This segment has the highest volume of currently-escalated requests (FAQ bot can't access savings plan data), directly maps to the "client-specific data + action execution" vision described by the team, and the client context layer built for this use case is reusable for all other segments.

Why the top risks are addressable

Compliance (Score 15): Scalable has already started feeding client-specific data to the LLM (confirmed in interview). The expansion to savings plan data is incremental, not a paradigm shift. Start with read-only access as a compliant first step.

Hallucination (Score 15): Unlike free-text retrieval, savings plan data is structured (amount, date, status, cash balance). Structured data injection has dramatically lower hallucination risk than unstructured RAG. Add confidence thresholds and auto-escalation as safety nets.

The Idea

Smart Support Agent: Architecture

Four capability layers: client knowledge, proactive diagnosis, confirmed actions, and intelligent escalation. Built as a platform service reusable across all product teams.

Layer 1: Knows the Client

Accesses portfolio positions, savings plan config, cash balance, transaction history. Data is injected as structured context, not retrieved via RAG.

Layer 2: Diagnoses Proactively

Identifies root cause of issues by cross-referencing client data. "Your savings plan failed because your cash was €57.70 short."

Layer 3: Takes Actions

Executes pre-approved actions (reschedule, adjust amount) with explicit client confirmation. Every action has an audit trail.

Layer 4: Escalates Intelligently

When confidence is low or the issue is complex, hands off to human agents with full conversation context and client data summary.

Task 2 · Evaluation Framework

How to measure AI quality before and after release.

Three evaluation layers: offline testing before release, online monitoring in production, and compliance-specific checks for regulatory adherence.

Layer 1: Offline Evaluation (Pre-Release)

Golden test set: 500+ labeled support cases with known-correct answers. Run on every model or prompt change.

Metrics: Answer accuracy (F1 score), hallucination rate (% of responses with fabricated info, measured by automated fact-checking against source data), retrieval quality (RAGAS: context relevance, faithfulness, answer relevancy), action safety (% of proposed actions matching expected).

Methodology: Automated regression suite. Hallucination detection via structured data comparison — if the agent says "your balance is €X" but the source data says €Y, that's a flagged hallucination. Red-team testing for adversarial prompts.

Layer 2: Online Evaluation (In Production)

Resolution rate: % of conversations resolved without human escalation. Current: 70%. Target: 85%.

CSAT: Post-conversation satisfaction (thumbs up/down + optional comment). Track per segment.

Action completion rate: % of AI-proposed actions that clients confirm and complete. Target: >40%.

Trace analysis: Weekly review of failed/escalated conversations. Identify systematic gaps — new FAQ topics, edge cases, prompt improvements needed.

Layer 3: Compliance-Specific Evaluation

Data leakage score: Automated checks that no PII or financial data from Client A appears in Client B's conversation.

Regulatory boundary: % of responses correctly refusing investment advice when asked. Target: 100%. Tested with adversarial prompts ("What stocks should I buy?").

Bias testing: Differential response quality across user demographics (language, account size, account age). Flagged if resolution rate varies >5% between cohorts.

Task 2 · Guardrails

Four defense layers against hallucination and data leakage.

Compliance is a design constraint, not a gate at the end. Every response passes through input sanitization, processing constraints, output classification, and immutable audit logging.

1

Input Guardrails

Prompt injection detection (block adversarial inputs). Topic classification: route to support, investment (Insights), or reject. PII detection and masking before LLM processing. Rate limiting per session.

2

Processing Guardrails

System prompt with hard constraints: no investment advice, no data from other clients, no actions without confirmation. Retrieval scope: model can only access data for the authenticated, consenting client. Action allowlist: only pre-approved operations (reschedule, adjust amount, escalate).

3

Output Guardrails

Response classifier checks every output for: hallucinated financial facts, investment recommendations, cross-client data leakage. Confidence threshold: responses below 0.80 confidence trigger automatic escalation to human agent. Mandatory disclaimers on any financial topic.

4

Observability & Audit

Full conversation and decision trace logging (immutable, append-only). Real-time monitoring dashboard: hallucination spikes, escalation surges, latency anomalies. Quarterly model audit report for BaFin readiness and EU AI Act high-risk classification compliance.

Implementation Roadmap

From POC to production in three phases.

A phased approach that validates the riskiest assumptions first, then scales. Compliance review is built into every phase, not added at the end.

Month 1-2

Phase 1: Internal Beta

Deploy to 5-10 support agents as a copilot (AI drafts responses, humans review before sending). Validate accuracy on real client cases. Measure hallucination rate on structured data. Get DPO/Legal sign-off on data access scope.

Exit criteria: Hallucination rate <1% on golden test set. Legal approval for read-only client data access.

Month 3-4

Phase 2: Limited Client Rollout

Roll out to 10% of clients (savings plan segment). AI handles diagnosis directly; actions still require human confirmation. Monitor resolution rate, CSAT, and escalation quality. A/B test against current FAQ bot.

Exit criteria: Resolution rate improvement of +10pp vs. control. CSAT equal or better than human-only support.

Month 5-6

Phase 3: General Availability + Actions

Full rollout to all clients. Enable AI-executed actions with client confirmation (reschedule, adjust). Expand to Segment 2 (traders) and Segment 3 (onboarding). Share client context layer as platform service for other teams (Insights, fraud).

Exit criteria: 85% resolution rate. Action adoption >40%. Platform API consumed by at least one other product team.
